# T1.7 — Evidencia ejecutable del workflow (iteración 2 sandbox)

> **Scope**: solo fase de planeación (5 invocaciones speckit-*). Sin `/speckit-implement`.
> **Caso de prueba**: "Build a simple todo CRUD with user authentication using Supabase + Vercel".
> **Fecha**: 2026-05-21 (sesión PM/continuación)
> **Propósito**: input para EVALUATION.md v0.2 (T1.8) y ADR-009 v0.5 PRELIMINAR (T2).

---

## Resumen ejecutivo

Las 5 skills de planeación de Spec Kit (`speckit-constitution`, `speckit-specify`, `speckit-clarify`, `speckit-plan`, `speckit-tasks`) ejecutaron end-to-end produciendo **6 artefactos** (`spec.md`, `checklists/requirements.md`, `plan.md`, `research.md`, `data-model.md`, `contracts/api.md`, `quickstart.md`, `tasks.md` — 47 tareas total). El stack **orquesta el proceso SDD limpio y ordenado**, pero **NO ejecuta reglas A* automáticamente** — la cobertura A1-A25 fue inyectada por el operador (yo) al rellenar plan.md/research.md/data-model.md/contracts/api.md/tasks.md. Sin ese overlay manual, el output sería un plan SDD genérico técnicamente correcto pero ciego a las reglas críticas (A5, A12, A20, A21, A22, A24).

**Insight Decisión A confirmado a nivel ejecutable**: las skills speckit-* aportan **estructura/proceso** (el "qué" y "cuándo" de cada artefacto), no **contenido arquitectónico** (las reglas A1-A25). El overlay declarativo del Framework es la **única fuente** del contenido A*.

**Hit rate del a-priori**: 4/4 categorías de skill speckit-* validadas como "cobertura A* = ninguna directa". La matriz preliminar (EVALUATION v0.1) estaba **correcta a priori** para Spec Kit.

### Cobertura A* — **0 reglas directas**, pero **fuerza A7 estructuralmente**

## Invocación 4 — `/speckit-plan`

### Inputs
- Spec.md y constitution.md
- `.specify/scripts/powershell/setup-plan.ps1 -Json` resolvió IMPL_PLAN + copió template a `specs/001-todo-management/plan.md`

### Mecánica observada
1. Setup script copió template (no NEEDS CLARIFICATION inicial — el operador es libre de marcarlos)
2. Phase 0 research → escribí 10 decisiones técnicas + 3 deferred (research.md)
3. Phase 1 design → escribí data-model.md (3 tablas + RLS + indexes + state transitions) + contracts/api.md (10 server actions con Result<T>) + quickstart.md (8 secciones de setup + smoke check)
4. Update CLAUDE.md SPECKIT markers → reemplazado contenido genérico con links a plan + spec + research + data-model + contracts + quickstart

### Resultado empírico — **CRÍTICO PARA ADR-009**

**Constitution Check obligatorio en el template** fue el momento donde **el overlay A* aterrizó al plan**. El template tiene la sección "Constitution Check" con la línea: *"[Gates determined based on constitution file]"*. Esto **es el punto de fricción más alto** del workflow speckit-*:

- Si el operador IGNORA esta sección o la rellena con bullet points genéricos → cobertura A* = 0 en el plan
- Si el operador la rellena CONSCIENTE del Framework → cobertura A* = 25/25 en el plan (como yo lo hice — 22 ✅ + 3 deferred)
- Skill **no fuerza** consultar constitution.md programáticamente

### Cobertura A* — **0 reglas directas**, pero **abre la puerta a 25 vía Constitution Check**

La diferencia entre un plan "vibe coder" y un plan "harness anti-alucinación" se decide en esta sección. La skill provee el slot estructural pero **no rellena el contenido**.

Mapping observado en mi plan.md (Constitution Check):

| Categoría de cobertura | Reglas | Plan-level approach |
|---|---|---|
| Direct + verificable | A1, A3, A5, A7, A11, A12, A14, A15, A20, A22 | RLS, hexagonal, zod, structured actions, adversarial tests |
| Direct + delegable a stack | A16, A17, A23 | Upstash rate limit + Vercel WAF + Vercel preview |
| Direct + parcialmente cubierto | A21 | logs ✅ metrics ✅ traces ❌ |
| Direct + diferido | A18, A19 | N/A v1 + deuda v1.1 |
| Direct + estructural en datos | A4, A6, A8, A13, A24 | Hexagonal imports, audit log, optimistic concurrency, soft-delete window |
| Direct + cultural | A2, A9, A10, A25 | Encapsulación, stop conditions, test isolation, owner-only model |

**44% Direct stack-executable + 56% via plan-level discipline**. Coincide con la matriz a-priori de EVALUATION v0.1.

### Gap observado
- Constitution Check sin enforcement: skill no aborta si el operador la deja vacía o vaga
- Skill sigma propuesta: **`sigma:enforce-constitution-check`** (post-procesador que valida que cada regla del constitution.md tiene una línea concreta en plan.md, o que se justifica con ADR)

## Conclusiones para ADR-009

### Veredicto del workflow speckit-* (fase planeación)

✅ **Spec Kit como orquestador del proceso SDD es ACEPTABLE** — produce artefactos coherentes, orden ejecutable, parallelización clara.

🟡 **Spec Kit como ejecutor de reglas A* es INSUFICIENTE** — su taxonomía cubre ~10/25 reglas, su Constitution Check no fuerza enforcement, su mapping FR/SC↔A* es manual.

✅ **El overlay declarativo del Framework (A1-A25) sigue siendo única fuente de verdad arquitectónica** — sin él, los outputs speckit-* son técnicamente correctos pero arquitectónicamente ciegos.

### Implicaciones operativas

1. **Adoptar Spec Kit para estructura/proceso** ✅ — la cadena specify→clarify→plan→tasks es valiosa
2. **Construir 3-5 skills sigma como post-procesadores/enforcement layer** sobre Spec Kit:
   - `sigma:operationalize-constitution` (sync Framework → constitution.md)
   - `sigma:enforce-constitution-check` (validar plan.md tiene línea concreta para cada A*)
   - `sigma:annotate-spec-with-a-rules` (mapping FR/SC ↔ A*)
   - `sigma:annotate-tasks-with-a-rules` (tag [A:N,M] en tasks)
   - `sigma:clarify-with-a-rules-taxonomy` (taxonomía clarify expandida a 25 cats)
3. **Documentar el "manual injection point"** del Constitution Check como momento crítico del workflow (entrenamiento del operador)

### Refinamiento de la matriz A* vs stack (input para T1.8)

| Bloque | Reglas | Cobertura a-priori (v0.1) | Cobertura ejecutable observada (v0.2) | Cambio |
|---|---|---|---|---|
| `speckit-*` planning skills | A7 indirecto, sin más | 0 directas | 0 directas, 1 indirecta (A7) | Sin cambio |
| Constitution Check del plan template | abre puerta a 25 | no en matriz original | gateway opcional a 25 | **NUEVO** |
| Tasks transitive coverage | depende del plan | no en matriz original | hereda del plan completamente | **NUEVO** |

### Hallazgos puente para iteración 3 (próxima sesión con T1.9 + /implement)

- **Tests adversariales en tasks** (T038-T041) son gates verificables — sirven como evidencia ejecutable de A5 + A12 cuando se ejecuten en /implement
- **RLS policies en migration** (T008) son artefacto verificable de A5 — Postgres rechaza queries cross-tenant
- **Adversarial CI gate** (T041) institucionaliza A15 (Unhappy Path First) en pipeline
- Comparación sdd-* vs speckit-* (T1.9, diferido a próxima sesión) debe replicar este mismo caso con el otro stack y comparar friction points en cada uno
