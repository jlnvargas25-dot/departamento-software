## 1. Resumen ejecutivo

Ambos workflows producen artefactos de planning utiles sobre el mismo caso, **pero con perfiles complementarios distintos**:

- **speckit-***: estructura **flujo-orientada** (constitution -> specify -> clarify -> plan -> tasks), un artefacto por fase, monolitico por dominio. Mejor para arrancar rapido con un caso bien delimitado. **Constitution Check es injection point manual del operador** (DEUDA-CONSTITUTION-CHECK-ENFORCEMENT confirmada empiricamente en T1.7).
- **sdd-***: estructura **artefacto-orientada** (explore -> proposal -> spec + design -> tasks), un artefacto por capacidad, capability-decomposed. Mejor para sistemas multi-dominio con boundaries explicitos. **A-rules embebidas inline** en specs/design/tasks por defecto (proxy de enforcement sin skill sigma adicional).

Conteo de output:

| Workflow | Files | Lines | KB | Tareas finales |
|---|---|---|---|---|
| speckit-* | 8 | 928 | ~70 | 47 |
| sdd-* | 7 | 2378 | ~156 | 47 |

**Coincidencia notable**: ambos workflows convergen en **47 tareas** sobre el mismo caso (case dimension drives task count, no el workflow).

**Decision D recomendada**: ver Seccion 6 — hibrido segun criterios explicitos.

## 3. Findings cualitativos

### F1: sdd-explore tools mismatch (gap operativo)
- **Observado**: el sub-agent `sdd-explore` no tiene tool Write. Produjo el contenido inline y devolvio handoff al orquestador para persistir.
- **Implicacion**: el operador debe escribir el archivo manualmente, o el sub-agent requiere wrapper.
- **DEUDA candidata**: `DEUDA-SDD-EXPLORE-NO-WRITE` o `DEUDA-SUB-AGENT-TOOL-MISMATCH-AUDIT`.

### F2: sdd-* save-to-engram-by-default override
- **Observado**: a pesar de instruccion explicita "openspec only — DO NOT save to engram", el sub-agent `sdd-explore` guardo en engram igual ("skill mandatory rule overrides run preference").
- **Implicacion**: las instrucciones de skill son mas pesadas que las instrucciones del orquestador en el contrato actual.
- **Mitigacion**: para fairness en T1.9, descartado el engram save (anotado en exploration.md).

### F3: A-rules surface emerges differently
- **speckit-***: A-rules aparecen al nivel del plan (Constitution Check) — un slot explicito, manual.
- **sdd-***: A-rules aparecen inline en specs ("(A5)", "(A12 ZT-3)" intercalados en scenarios) y en cada ADR del design.
- **Hipotesis (N=1)**: el operador inyecta A-rules con menos friccion en sdd-* porque la estructura las invita por capa; en speckit-* requiere un acto consciente.

### F4: Strict TDD propagation
- **speckit-***: no detecto test runner instalado, marca como deudas pero igual incluye test tasks (siempre tests, por decision del operador en T1.7).
- **sdd-***: openspec/config.yaml `strict_tdd: false` propaga a `sdd-apply`/`sdd-verify` para no exigir TDD si no hay runner; tasks phase incluyo install de Vitest/Playwright como Phase 0.
- **Ganador**: sdd-* tiene canal formal (config.yaml strict_tdd boolean); speckit-* depende del operador.

### F5: Capability boundary in specs
- **speckit-***: una sola spec.md con 3 user stories anidadas (US1 todos, US2 auth, US3 adversarial integration).
- **sdd-***: 3 specs separadas por dominio (auth, todos, observability). Adversarial scenarios estan dentro de cada spec, no en una US3.
- **Implicacion arquitectonica**: sdd-* alinea naturalmente con A20 hexagonal (un dominio = una spec = un puerto + adapters); speckit-* requiere mas re-mapping.

### F6: Dependency graph + parallel ops
- **speckit-* tasks.md**: Phase A/B/C listing + dependencias en prosa.
- **sdd-* tasks.md**: DAG Mermaid + Parallel-OK groups + bottleneck flagging (T-08 RLS migration flagged como hard bottleneck).
- **Implicacion**: sdd-* da material directamente accionable por un orquestador como `/multi-execute`; speckit-* requiere parseo manual.

### F7: Review workload forecast
- **speckit-***: no incluido nativamente (deferido a `speckit-analyze` opcional).
- **sdd-***: incluido en tasks.md (estimacion ~1800-2400 lineas, single-PR con size:exception para sandbox).
- **Implicacion**: sdd-* esta tooled para el caso "fresh review rule" del orchestrator CLAUDE.md.

### F8: Reversibility (R-5)
- **speckit-***: implicit. data-model menciona soft delete, no hay reversal plan formal.
- **sdd-* design.md**: seccion explicita "Reversibility" listando cada operacion destructiva (delete, purge, account-delete) con plan de rollback.
- **Ganador**: sdd-*.

### F9: Cost
- **speckit-***: orquestado por scripts powershell pequeños + Skill tool directo; ~25-35k tokens estimados.
- **sdd-***: 4 sub-agents (explore, propose, spec, design, tasks) cada uno con context fresh; ~120-150k tokens estimados.
- **Implicacion**: sdd-* es 3-5x mas caro en planning. Justificado SI: necesitas capability decomposition + DAG explicito + 8 ADRs. NO justificado SI: caso simple con un solo dominio.

### F10: Convergencia 47 tasks
Ambos workflows produjeron **exactamente 47 tasks** sobre el mismo caso. La granularidad viene del case dimension, no del workflow. Implicacion: a la hora de calibrar `/multi-execute` o `/sdd-apply`, el budget de tasks por feature es predecible por dominio, no por skill.

## 5. Lecciones empiricas T1.9

### LECCION 24 candidata (N=1, 2026-05-21 PM continuacion 2) — Convergencia 47 tasks
Ambos workflows produjeron **exactamente 47 tasks** sobre el mismo caso. La granularidad de tasks viene del case dimension, no del workflow. Implicacion: a la hora de calibrar `/multi-execute` o `/sdd-apply`, el budget de tasks por feature es predecible por dominio, no por skill.

### LECCION 25 candidata (N=1) — speckit-* arranca rapido, sdd-* documenta mas
speckit-* genera un primer artefacto util en ~15 s; sdd-* tarda ~45 s para inicializar openspec/ + escribir exploration.md. **Pero** sdd-* produce 2.5x mas contenido (2378 vs 928 lineas). Es un tradeoff de speed-to-first-artifact vs depth-of-planning. Justifica un workflow hibrido (speckit-* para tarjetas chicas, sdd-* para features con multiple capacidades).

### LECCION 26 candidata (N=1) — sub-agent tools mismatch detectable a-priori
El sub-agent `sdd-explore` no tiene Write tool, pero el skill esperaba persistir. Esto se puede detectar en la fase de skill-resolver del orquestador (auditando tool surface vs skill contract). DEUDA candidata: `DEUDA-SDD-EXPLORE-NO-WRITE` o `DEUDA-SUB-AGENT-TOOL-MISMATCH-AUDIT`.

### LECCION 27 candidata (N=1) — skill mandatory rule overrides run preference
El sub-agent `sdd-explore` guardo en engram a pesar de instruccion explicita "openspec only". Esto sugiere que ciertos comportamientos de skill no se pueden override desde el orquestador. Implicacion para Framework: las skills sigma futuras deben respetar instrucciones del orquestador para no romper isolation en sandboxes/comparaciones.

### LECCION 28 candidata (N=1) — capability decomposition emerge del workflow
sdd-* produjo 3 specs aisladas por dominio (auth/todos/observability) sin que el operador lo pidiera explicitamente — emergio del config.yaml `domains:` block. speckit-* produjo 1 spec.md monolitica con 3 user stories anidadas. **Implicacion**: el config.yaml `domains:` block de openspec acts as a structural prior que sesga sdd-* hacia hexagonal A20 sin friccion. speckit-* no tiene equivalente estructural — el operador debe hexagonalizar manualmente.

## 6. Decision D — Resolucion

### Opciones del addendum SIGUIENTE-SESION
- **(i) Spec Kit como backbone** — workflow flujo-orientado
- **(ii) SDD alternativo (sdd-*)** — workflow artefacto-orientado
- **(iii) Combinar segun fase del workflow** — hibrido

### Recomendacion: **(iii) Combinar** con criterios explicitos

| Cuando usar | speckit-* | sdd-* |
|---|---|---|
| Caso simple con 1 dominio claro | X | |
| Need speed-to-first-artifact (<30s) | X | |
| Caso con 2+ capacidades / boundaries explicitos | | X |
| Necesitas DAG explicito para `/multi-execute` | | X |
| Necesitas ADRs estructurados en design | | X |
| Operador no instruido del Framework (cobertura A* fallback) | | X |
| Operador instruido del Framework + costo importante | X (cobertura 76% suficiente) | |
| Multi-tenant strict isolation (A5) critico | | X (specs aislan dominio) |

**Backbone para iteracion 3 y futuro**:
- **Default**: speckit-* por velocidad y cost (lo que ya hicimos en T1.7).
- **Escalation**: sdd-* cuando el case requiere capability decomposition o cuando el operador no esta instruido del Framework (sdd-* tiene mas A-rule injection nativa).
- **Refinamiento**: las 4 sigma MVP skills planeadas en ADR-009 (`operationalize-constitution`, `enforce-constitution-check`, `multi-tenant-isolation-checker`, `dependency-cycle-detector`) cierran los gaps de speckit-* (Constitution Check enforcement + adversarial fairness). Con esas 4 skills, speckit-* se acerca a la cobertura de sdd-* a fraccion del costo.

### Implicacion para ADR-009 v0.5 -> v1.0

- **Status promoter**: PROPOSED -> **PROPOSED-ACTUALIZADO** (Decision D resuelta, pero promocion final a ACCEPTED requiere iteracion 3)
- **Nuevas evidencias**:
  - speckit-* y sdd-* convergen en 47 tasks -> case dimension drives count (Leccion 24)
  - sdd-* tiene mejor structural A* enforcement -> justifica sigma:enforce-constitution-check para llevar speckit-* a paridad
  - sdd-* es 3-5x mas caro en tokens -> cost-aware default sigue siendo speckit-*
  - sdd-* tiene capability decomposition emergent (Leccion 28) -> justifica sdd-* para multi-domain features
- **Decision D resuelta**: backbone hibrido — speckit-* default + sdd-* escalation segun matriz F1-F10.
- **Pendiente para v1.0 final**: iteracion 3 (/speckit-implement) para medir cobertura ejecutable real post-implement.
