> **Fecha**: 2026-05-21 noche (cierre #3) | **Versión**: 0.1 LITE
> **Scope reducido**: lectura de las 9 SKILL.md de sdd-* + comparación a-priori contra speckit-* (cuyo workflow ejecutable está documentado en `T1.7-EVIDENCE.md`).
> **Asimetría declarada**: speckit-* fue **ejecutado** sobre caso real (`specs/001-todo-management/` + 47 tasks). sdd-* fue **leído** sin ejecutar. Decisión D queda PRELIMINAR; ejecución empírica de sdd-* sobre el mismo caso queda como deuda para iteración 3.

## Resumen ejecutivo

**Recomendación preliminar Decisión D**: `sdd-*` como **backbone primario** del Framework, `speckit-*` como **fallback ligero** para features triviales sin necesidad de versionado de behavior. Sub-decisiones:

1. **`sdd-*` gana** en: rigor arquitectónico, alineación con 7 principios del Framework, enforcement TDD, manejo de evolución (delta specs + archive merge), Review Workload Forecast (chained PRs), persistence flexible (4 modos), sub-agent delegation con context isolation.
2. **`speckit-*` gana** en: velocidad de arranque, simplicidad operativa, costo en tokens, granularidad de checklists nativos.
3. **Ambas son complementarias**: el usuario puede elegir según la criticidad de la feature.

**Validación pendiente**: ejecutar sdd-* sobre el mismo caso "todo+auth Supabase" para comparar evidencia ejecutable lado a lado (iteración 3 o cuarto día).

## Tabla comparativa estructural

| Dimensión | `speckit-*` | `sdd-*` | Ventaja |
|---|---|---|---|
| **Modelo de invocación** | Skills locales `user-invocable: true` | `delegate_only: true` + `user-invocable: false` + ORCHESTRATOR GATE en cada SKILL.md | sdd-* fuerza context isolation vía sub-agent delegation |
| **Cantidad de fases** | 9 (constitution, specify, clarify, plan, tasks, implement, analyze, checklist, taskstoissues) | 9 (init, explore, propose, spec, design, tasks, apply, verify, archive) | Paridad cuantitativa, diferencia cualitativa |
| **Persistence backend** | File-based exclusivo (`.specify/` + `specs/{feature}/`) | 4 modos: `engram` / `openspec` / `hybrid` / `none` | sdd-* más flexible (solo + team + cross-LLM) |
| **Unit of work** | "Feature" (independent) | "Change" (versionado, con delta + archive) | sdd-* maneja mejor evolución de behavior |
| **Delta specs** | No existe | sdd-spec produce delta specs; sdd-archive funde delta→main spec | sdd-* permite cambios incrementales sobre specs vivos |
| **Archive merge** | No existe | sdd-archive: `ADDED → append`, `MODIFIED → replace`, `REMOVED → delete` por requirement | sdd-* mantiene specs vivas evolutivas |
| **Constitution Check** | Slot manual en `plan-template.md` (LECCIÓN 22: "manual injection point") | Sin slot explícito en sdd-design SKILL.md visible; pendiente revisar `references/` | speckit-* explícito, sdd-* implícito vía proposal |
| **Review Workload Forecast** | No existe | sdd-tasks lo incluye obligatorio (`400-line budget risk`, `Chained PRs recommended`, `Delivery strategy`, `Chain strategy`) | sdd-* tiene política de PR-size + chained PR strategy nativa |
| **Strict TDD enforcement** | Template TDD presente, no enforce | sdd-init detecta `strict_tdd` + propaga a apply/verify con `STRICT TDD MODE IS ACTIVE` flag | sdd-* enforce verificablemente |
| **Verify phase** | `speckit-analyze`: cross-artifact consistency, NO ejecuta tests | `sdd-verify`: Hard Rule "static analysis alone is never verification"; corre tests reales, mapea spec scenario ↔ test pasado | sdd-* gate de calidad ejecutable |
| **Persistence de progress** | tasks.md con `[x]` marks | apply-progress como observación separada + tasks.md con `[x]` (modo hybrid) + `mem_update` por task | sdd-* tiene continuidad cross-batch via mem_search |
| **Output envelope** | Outputs sin contrato formal entre fases | Estructurado: `status / executive_summary / artifacts / next_recommended / risks / skill_resolution` | sdd-* contratos inter-fase explícitos |
| **Orquestador** | Implícito (usuario) | Explícito: el orchestrator (Claude) controla delegation, resuelve TDD, inyecta compact rules de skill-registry | sdd-* asume harness anti-alucinación (alineado con visión Julián) |

## Match con 7 principios del Framework

| Principio | `speckit-*` | `sdd-*` | Comentario |
|---|---|---|---|
| 1. Python traza → IA recorre → Python verifica | 🟡 templates Python opcionales | 🟢 sdd-init detecta capabilities + sdd-verify ejecuta tests deterministically | sdd-* más Python-traced |
| 2. 3 capas PREVENTIVA → VERIFICABLE → CORRECTIVA | 🟡 spec+plan = preventiva; analyze = verificable; sin correctiva | 🟢 proposal+spec+design = preventiva; verify = verificable; archive merge + delta evolution = correctiva implícita | sdd-* encarna las 3 capas |
| 3. Dominio-first | 🟢 specify NL→spec | 🟢 explore + propose dominio antes de spec | Paridad |
| 4. Auto-fix > finding | 🔴 Ninguno explícito | 🔴 Ninguno explícito (sdd-verify dice "Do not fix issues; report them") | Ambos requieren overlay externo |
| 5. Polinización cruzada | 🔴 features aisladas | 🟢 archive merge a main specs polinizable | sdd-* permite polinización entre changes |
| 6. Descubrir antes de ejecutar | 🟡 clarify es subset | 🟢 explore es fase dedicada con investigación de codebase | sdd-* explícito |
| 7. Meta-producto recursivo | 🟡 aplicable | 🟢 aplicable, además sdd-* puede gestionar a sí mismo via change folder | sdd-* meta-recursivo mejor encajado |

**Score**: speckit-* 2🟢 4🟡 1🔴 ; sdd-* 6🟢 1🟡 1🔴 ⇒ **sdd-* alineamiento ~3x mayor**.

## Decisión D PRELIMINAR

**Backbone primario propuesto**: `sdd-*` (Gentle AI Programming).
**Backbone secundario**: `speckit-*` (GitHub Spec Kit) — para features triviales, exploración rápida, prototipos.

**Justificación**:
1. **Match con Framework**: 6/7 principios encajan 🟢 (vs 2/7 de speckit-*).
2. **Persistence flexibility**: 4 modos cubre solo-dev (engram) + team (openspec) + cross-LLM (hybrid).
3. **Delta + archive**: especificación viva evolutiva, evita feature-creep.
4. **Workload Forecast**: política de PR-size + chained PR nativa (alineada con DEUDA-PROTOCOLO-CIERRE-PR-SIZE futura).
5. **TDD enforcement**: cierra LECCIÓN 22 (manual injection point) parcialmente.
6. **Sub-agent delegation**: cada fase corre con context fresh + compact rules del skill-registry → menos drift (LECCIÓN 16/24).

**Riesgo**: la evidencia es **a-priori**. La iteración 3 ejecutable (caso "todo+auth Supabase" via sdd-*) puede revelar fricciones que cambien la decisión.

**Acción próxima**: iteración 3 ejecutable sobre el mismo caso para promover ADR-009 v0.5 → v1.0 ACCEPTED o REJECTED.
